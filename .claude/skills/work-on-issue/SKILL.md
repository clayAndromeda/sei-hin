# Issue駆動開発ワークフロー

GitHub Issueを読み、作業計画を立て、ユーザーの承認を得てから実装する一連のルーチンを実行する。

## 手順

### 1. Issue確認

引数としてIssue番号が渡された場合（`$ARGUMENTS`）:
- `gh issue view $ARGUMENTS` でIssue詳細を確認する

引数が空の場合:
- `gh issue list` でオープンなIssue一覧を表示する
- ユーザーにどのIssueに取り組むか確認する

### 2. 関連情報の収集

- Issue本文だけでなく、コメントも確認する（`gh issue view <番号> --comments`）
- 関連するIssueがあれば、それも確認する
- 対象となるコードベースを調査し、影響範囲を把握する

### 3. 作業計画の立案

以下の項目を含む作業計画をユーザーに提示する:

- **対象Issue**: Issue番号とタイトル
- **概要**: 何を実装/修正するか
- **影響範囲**: 変更が必要なファイル・コンポーネント
- **実装ステップ**: 具体的な作業手順（番号付きリスト）
- **テスト方針**: どのようにテストするか
- **懸念事項**: あれば記載

### 4. ユーザーの承認を待つ

計画を提示したら、**必ずユーザーの承認を待つこと**。承認なしに実装を開始しない。
ユーザーからのフィードバックがあれば計画を修正する。

### 5. 実装

承認を得たら、計画に沿って実装を進める。

- TodoWriteで作業項目を管理する
- 各ステップを完了するごとにTodoを更新する
- テストを書き、`npm test` で動作確認する

### 6. コミット前の確認

実装完了後、コミット前に以下をユーザーに提示して承認を得る:

- 変更したファイルの一覧
- 変更内容の要約
- 予定するコミットメッセージ（`<type>: <概要> (#<Issue番号>)` 形式）

ユーザーの承認を得てからコミットを実行する。

### 7. Issue更新

コミット後:
- 完了した場合: `gh issue close <番号> -c "実装完了しました"`
- 追加対応が必要な場合: `gh issue comment <番号> -b "コメント内容"` で状況を報告
